## 1. 产品概述

### 1.1 背景

当前销售体系以阶段表格方式管理客户、商机、周进展、方案、合同、开票和回款信息。该方式适合早期梳理业务，但在项目型大客户销售场景下，会出现信息分散、过程不可追踪、关系经营缺少结构化沉淀、周进展重复录入、合同开票回款难以闭环、管理层预测和风险判断滞后的问题。

本产品将销售体系升级为项目型大客户CRM，围绕客户经营和商机生命周期建立统一业务系统，并引入AI销售助手作为信息录入员工，降低一线业务人员录入成本。

### 1.2 产品定位

项目型大客户CRM是面向政企、产业客户、项目制销售团队的销售过程管理与经营分析系统。系统覆盖客户、联系人、商机、销售行动、方案标书、合同、开票、回款、经营驾驶舱和AI销售助手十个模块。

### 1.3 产品目标

- 统一管理客户、联系人、干系人和客户关系地图。
- 管理商机从商业线索到商业成交的完整生命周期。
- 将销售行动独立为过程记录入口，并自动汇总商机周进展。
- 管理方案、标书、报价、利润测算、内部评审和客户反馈。
- 打通合同、开票、回款和发票核销闭环。
- 为管理层提供销售漏斗、预测、风险、合同、开票、回款看板。
- 通过AI销售助手支持文本、纪要、附件等信息的智能识别、草稿生成和确认入库。

## 4. 核心业务流程

### 4.1 主业务流程

客户建档 -> 联系人/干系人维护 -> 商机创建 -> 商机阶段推进 -> 销售行动记录 -> 周进展自动汇总 -> 方案与标书协同 -> 商业谈判 -> 商机成交 -> 合同创建 -> 开票计划/开票登记 -> 回款流水 -> 发票核销 -> 经营驾驶舱分析

### 4.2 AI录入流程

业务人员提交信息 -> AI识别业务对象 -> AI匹配客户/商机 -> AI提取字段 -> 缺失信息追问 -> 生成待确认草稿 -> 人工确认 -> 写入CRM -> 生成写入日志

### 4.3 回款核销流程

合同生成回款计划 -> 财务登记回款流水 -> 财务选择待核销发票 -> 创建发票核销明细 -> 系统校验金额 -> 财务确认核销 -> 更新发票、回款流水、合同回款金额

### 4.4 登录与权限流程

用户登录 -> 系统校验账号状态和密码 -> 建立会话 -> 加载角色权限和数据范围 -> 用户访问菜单或数据 -> 系统校验操作权限和数据权限 -> 记录关键操作审计日志

## 5. 功能需求

### 5.0 系统基础与权限管理

#### 5.0.1 功能说明

系统基础与权限管理为CRM提供账号认证、组织用户、角色权限、数据范围、字典配置和审计能力。该模块是所有业务模块运行的基础，不直接承载销售业务，但决定用户能否登录、能看到哪些菜单、能操作哪些数据、关键操作是否可追溯。

| SYS-002 | 登出 | P0 | 支持用户主动登出，服务端会话失效 |
| SYS-003 | 会话管理 | P0 | 支持登录态、会话过期、异地登录策略、强制下线 |
| SYS-004 | 修改密码 | P0 | 登录用户可修改密码，需校验旧密码和密码强度 |
| SYS-005 | 忘记密码/重置密码 | P0 | 支持管理员重置或通过令牌重置密码 |
| SYS-006 | 用户管理 | P0 | 创建、编辑、启用、停用用户，绑定部门和角色 |
| SYS-007 | 组织管理 | P0 | 维护部门、团队、上下级组织、销售区域 |
| SYS-008 | 角色管理 | P0 | 维护销售、销售负责人、售前、商务、财务、管理层、管理员等角色 |
| SYS-009 | 菜单权限 | P0 | 控制用户可见菜单和页面入口 |
| SYS-010 | 操作权限 | P0 | 控制新增、编辑、删除、审批、导出、核销、配置等动作 |
| SYS-011 | 数据权限 | P0 | 支持本人、本部门、本部门及下级、协同数据、全局等范围 |
| SYS-012 | 字典配置 | P0 | 配置客户等级、商机阶段、商机状态、行动类型、风险类型等枚举 |
| SYS-013 | 登录日志 | P1 | 记录登录成功、失败、登出、锁定等事件 |
| SYS-014 | 操作审计日志 | P0 | 记录关键业务操作、金额变更、权限变更、AI写入等 |
| SYS-015 | 账号安全策略 | P1 | 支持密码有效期、失败锁定、最小长度、复杂度规则 |

#### 5.0.3 账号与密码规则

- 用户账号必须唯一。
- 停用用户不得登录系统。
- 密码必须加密存储，不允许明文保存。
- 修改密码需校验旧密码。
- 管理员重置密码后，用户首次登录应提示修改密码。
- 连续登录失败超过配置次数后，账号可临时锁定。
- 会话过期后需重新登录。

#### 5.0.4 权限规则

- 菜单权限决定用户能否看到模块入口。
- 操作权限决定用户能否执行新增、编辑、删除、审批、导出、核销、配置等动作。
- 数据权限决定用户能查看和操作哪些业务记录。
- 财务核销、合同终止、商机关闭、权限配置、AI关键写入等动作必须记录审计日志。
- 系统管理员可配置权限，但权限变更本身也必须审计。

#### 5.0.5 验收标准

- 用户可通过账号密码登录和登出。
- 停用用户无法登录。
- 用户可修改密码，管理员可重置密码。
- 不同角色登录后看到的菜单和可操作按钮不同。
- 销售个人只能访问本人或协同数据，销售负责人可访问本部门数据，管理层可访问全局数据。
- 关键操作能在审计日志中查询到操作人、操作时间、操作对象和变更内容。

- 商业谈判
- 商业成交

- 暂停
- 关闭
- 取消跟进

| OPP-002 | 阶段推进 | P0 | 支持按阶段规则推进 |
| OPP-003 | 状态管理 | P0 | 支持跟进中、暂停、关闭、取消跟进 |
| OPP-004 | 关闭/取消跟进 | P0 | 记录关闭类型、原因、说明、时间、操作人、是否可重启 |

#### 5.3.5 业务规则

- 商业线索阶段必须有客户、机会点、责任人。
- 商业验证阶段必须补充项目需求、预算、周期、采购流程。
- 商业方案阶段必须有技术方案状态、干系人方案状态、当前进展。
- 商业谈判阶段必须有报价/利润测算、标书自评、预计合同金额。
- 商业成交后应生成或关联合同。
- 关闭或取消跟进商机不进入默认跟进列表，不计入销售预测。

#### 5.3.6 验收标准

- 商机阶段和状态可独立维护。
- 商机关闭或取消跟进时必须填写原因。
- 销售行动完成后，商机详情能展示周进展汇总。

| CTR-004 | 合同附件 | P0 | 管理合同正文、盖章版、补充协议 |
| CTR-005 | 合同变更 | P1 | 支持金额、范围、付款条件变更记录 |
| CTR-006 | 交付/验收节点 | P1 | 管理项目启动、交付、初验、终验、质保等节点 |
| CTR-007 | 生成开票计划 | P1 | 根据合同开票条件生成 |
| CTR-008 | 生成回款计划 | P1 | 根据合同付款条件生成 |

#### 5.6.3 验收标准

- 商机成交后可以创建或关联合同。
- 合同金额变更后，应提示同步调整开票和回款计划。
- 合同终止后，未执行计划标记为终止。

### 5.7 开票管理

#### 5.7.1 功能说明

开票管理基于合同和开票条件，管理开票计划、申请、审批、发票登记、签收、异常和已核销金额。

#### 5.7.2 功能清单

| 编号 | 功能 | 优先级 | 说明 |
|---|---|---|---|
| INV-001 | 开票计划 | P0 | 根据合同创建开票计划 |
| INV-002 | 开票申请 | P0 | 销售或商务发起开票申请 |
| INV-003 | 发票登记 | P0 | 财务登记发票号码、金额、日期、抬头 |
| INV-004 | 开票状态 | P0 | 计划中、审批中、已开票、已寄送、已签收、作废、异常 |
| INV-005 | 开票异常 | P0 | 记录抬头错误、税号错误、金额错误、客户拒收等 |
| INV-006 | 发票核销状态 | P0 | 展示已核销金额和未核销金额 |
| INV-007 | 开票看板 | P1 | 按客户、合同、负责人、月份统计 |

#### 5.7.3 验收标准

- 开票总额不能超过合同金额。
- 发票已开票后可被回款核销明细关联。
- 发票作废或异常时，应影响合同已开票金额统计。

### 5.8 回款管理

#### 5.8.1 功能说明

回款管理采用回款计划、回款流水、发票核销明细三层模型，支持回款与发票多对多核销。

#### 5.8.2 功能清单

| 编号 | 功能 | 优先级 | 说明 |
|---|---|---|---|
| PAY-001 | 回款计划 | P0 | 根据合同付款条件生成 |
| PAY-002 | 回款流水登记 | P0 | 财务登记到账金额、日期、流水号 |
| PAY-003 | 发票核销 | P0 | 通过核销明细匹配回款流水和发票 |
| PAY-004 | 多对多核销 | P0 | 一笔回款多张发票，一张发票多笔回款 |
| PAY-005 | 核销校验 | P0 | 回款核销不超过到账金额，发票核销不超过发票金额 |
| PAY-006 | 逾期预警 | P0 | 到期未回款或未核销预警 |
| PAY-007 | 回款跟进 | P1 | 逾期后生成销售行动或提醒 |
| PAY-008 | 回款看板 | P1 | 展示计划、到账、核销、逾期、应收余额 |

#### 5.8.3 验收标准

- 一笔回款流水可以拆分核销多张发票。
- 一张发票可以被多笔回款流水分次核销。
- 核销确认后，合同已回款金额、发票已核销金额、回款流水已核销金额同步更新。

### 5.9 经营驾驶舱

#### 5.9.1 功能说明

经营驾驶舱用于汇总客户经营、商机推进、销售行动、合同、开票、回款数据，支持管理层查询、分析、预警和追责。

#### 5.9.2 功能清单

| 编号 | 功能 | 优先级 | 说明 |
|---|---|---|---|
| DASH-001 | 管理层总览 | P0 | 漏斗、合同、开票、回款、风险 |
| DASH-002 | 销售漏斗 | P0 | 按阶段统计数量、金额、赢率 |
| DASH-003 | 商机预测 | P0 | 预计金额 x 赢率，按月份预测 |
| DASH-004 | 销售行动看板 | P0 | 行动数、完成率、逾期、风险 |
| DASH-005 | 合同看板 | P0 | 合同额、履约状态、风险 |
| DASH-006 | 开票看板 | P0 | 已开票、未开票、异常开票 |
| DASH-007 | 回款看板 | P0 | 已回款、逾期、应收余额 |
| DASH-008 | 数据下钻 | P1 | 从指标进入明细记录 |
| DASH-009 | 绩效排行 | P1 | 按负责人、部门统计商机、合同、回款 |

#### 5.9.3 验收标准

- 管理层能查看销售漏斗、合同额、开票额、回款额、逾期金额。
- 指标可按部门、负责人、客户、时间过滤。
- 风险和逾期指标可下钻到明细。

### 5.10 AI销售助手

#### 5.10.1 功能说明

AI销售助手作为统一信息入口和智能录入员工，帮助业务人员把碎片信息转成CRM草稿，经人工确认后写入系统。

#### 5.10.2 功能清单

| 编号 | 功能 | 优先级 | 说明 |
|---|---|---|---|
| AI-001 | 信息输入 | P0 | 支持文字、会议纪要、附件等录入 |
| AI-002 | 业务对象识别 | P0 | 识别客户、联系人、商机、销售行动、合同、开票、回款 |
| AI-003 | 字段提取 | P0 | 提取客户、商机、金额、日期、责任人、下一步等字段 |
| AI-004 | 已有记录匹配 | P0 | 判断新建还是更新已有记录 |
| AI-005 | 草稿生成 | P0 | 生成AI提取草稿 |
| AI-006 | 缺失信息追问 | P0 | 生成补充信息任务 |
| AI-007 | 人工确认入库 | P0 | 经确认后写入CRM |
| AI-008 | 销售行动自动生成 | P0 | 从沟通内容生成行动记录 |
| AI-009 | 风险识别 | P1 | 识别预算、关系、竞争、合同、回款风险 |
| AI-010 | 写入审计 | P0 | 保存原始输入、草稿、确认人、写入日志 |

#### 5.10.3 AI写入规则

- AI不直接静默写入关键业务数据。
- 客户、联系人、商机、销售行动可由AI生成草稿后确认写入。
- 合同、开票、回款必须由对应角色确认后写入。
- 低置信度、字段冲突、金额日期冲突必须进入确认任务。
- AI不能绕过字段必填、阶段规则、金额校验和权限控制。

#### 5.10.4 验收标准

- 用户输入一段销售沟通文本后，AI能生成销售行动草稿。
- AI能匹配已有客户和商机，并提示置信度。
- 用户确认后，销售行动写入CRM并更新商机周进展。
- 每次AI写入都有日志可查。

## 6. 权限需求

权限体系由四层组成：

- 认证权限：用户是否允许登录系统。
- 菜单权限：用户能看到哪些模块入口。
- 操作权限：用户能执行哪些动作，如新增、编辑、删除、审批、导出、核销。
- 数据权限：用户能访问哪些业务数据，如本人、本部门、本部门及下级、协同、全局。

| 权限对象 | 销售个人 | 销售负责人 | 售前/方案 | 商务/法务 | 财务 | 管理层 | 管理员 |
|---|---|---|---|---|---|---|---|
| 客户 | 本人/协同 | 本部门 | 关联可见 | 关联可见 | 关联可见 | 全局可见 | 全局管理 |
| 联系人 | 本人/协同 | 本部门 | 关联可见 | 关联可见 | 关联可见 | 全局可见 | 全局管理 |
| 商机 | 本人/协同 | 本部门 | 关联可见 | 关联可见 | 关联可见 | 全局可见 | 全局管理 |
| 销售行动 | 本人/参与 | 本部门 | 关联可见 | 关联可见 | 回款相关可见 | 全局可见 | 全局管理 |
| 方案与标书 | 关联可见 | 本部门 | 可维护 | 评审可见 | 只读 | 全局可见 | 全局管理 |
| 合同 | 关联可见 | 本部门 | 只读 | 可维护 | 关联可见 | 全局可见 | 全局管理 |
| 开票 | 关联可见 | 本部门 | 只读 | 关联可见 | 可维护 | 全局可见 | 全局管理 |
| 回款 | 关联可见 | 本部门 | 只读 | 关联可见 | 可维护 | 全局可见 | 全局管理 |
| 经营驾驶舱 | 个人 | 本部门 | 无 | 相关 | 财务 | 全局 | 全局 |
| AI销售助手 | 本人提交 | 本部门任务 | 关联任务 | 关联任务 | 财务任务 | 查看日志 | 全局管理 |

关键权限要求：

- 合同审批、合同终止、开票确认、回款确认、发票核销、核销撤销必须具备专项操作权限。
- 导出客户、联系人、商机、合同、开票、回款数据必须具备导出权限。
- 权限配置、角色配置、数据字典配置仅管理员可操作。
- AI销售助手可生成草稿和确认任务，但写入仍受当前确认人的业务权限约束。
- 数据权限以业务记录的负责人、归属部门、协同人员、参与人和角色范围共同判定。

## 7. 非功能需求

### 7.1 安全与审计

- 所有关键数据变更需记录操作人、操作时间、变更前后数据。
- AI写入必须可追溯到原始输入和确认人。
- 金额、合同、开票、回款、核销操作需要权限校验。
- 关闭商机、合同终止、核销撤销等动作需记录原因。

### 7.2 性能

- 列表页常规查询响应时间不超过3秒。
- 驾驶舱核心指标在常规数据量下响应时间不超过5秒。
- 支持按客户、商机、负责人、时间范围进行高频筛选。

### 7.3 可配置性

- 商机阶段、商机状态、客户等级、行动类型、风险类型、预警阈值应可配置。
- 角色权限和数据范围应可配置。
- AI助手的可写入对象、确认规则、置信度阈值应可配置。

### 7.4 数据质量

- 客户名称、商机名称、合同编号、发票号码、银行流水号应支持重复校验。
- 金额类字段使用精确小数。
- 客户、商机、合同、发票、回款之间的关联关系必须完整。

## 8. 指标口径

| 指标 | 计算口径 |
|---|---|
| 商机总金额 | 跟进中商机预计合同金额合计 |
| 加权预测金额 | 跟进中商机预计合同金额 x 赢率 |
| 成交合同额 | 已签署或履约中合同金额合计 |
| 已开票金额 | 已开票且未作废发票金额合计 |
| 已回款金额 | 已确认核销金额合计 |
| 应收余额 | 合同金额 - 已确认核销金额 |
| 开票率 | 已开票金额 / 合同金额 |
| 回款率 | 已确认核销金额 / 合同金额 |
| 逾期回款金额 | 超过计划回款日期且未完全回款的计划金额 |
| 行动完成率 | 已完成行动数量 / 计划行动数量 |
| 长期未跟进客户数 | 最近跟进时间超过配置天数的客户数 |

## 10. 风险与依赖

| 风险 | 影响 | 应对 |
|---|---|---|
| 一线销售不愿录入 | 数据不完整 | 通过AI销售助手和移动化录入降低成本 |
| 阶段规则过重 | 销售推进受阻 | 阶段准入规则先轻后重，可配置 |
| 财务核销逻辑复杂 | 回款数据不准确 | 采用核销明细中间表，财务确认后生效 |
| AI误写入 | 关键数据错误 | 草稿确认、低置信度拦截、写入日志 |
| 权限边界不清 | 数据泄露或协同受阻 | 按角色和数据归属配置权限 |
| 指标口径争议 | 管理看板不可用 | 指标口径在系统配置中固化并可审计 |

## 11. 验收总则

- 十个模块的核心对象、核心状态和关键关联关系应可用。
- 客户、商机、销售行动三者应能形成完整销售过程链路。
- 商机周进展必须由销售行动汇总生成，而不是依赖商机主表手工填报。
- 合同、开票、回款、核销应能形成财务闭环。
- 发票与回款必须支持多对多核销。
- AI销售助手必须采用草稿确认机制，并保留写入日志。
