## 1. 默认推荐方案

| 层级 | 默认选择 | 决策结论 | 主要理由 |
|---|---|---|---|
| 前端 | React + Ant Design | 启动默认采用 | 企业后台表格、筛选、表单、详情页和权限菜单较多，Ant Design生态成熟，适合快速搭建CRM管理端。 |
| 后端 | Java Spring Boot | 启动默认采用 | CRM涉及认证权限、事务、审计、数据权限、复杂业务规则和后续财务核销，Spring Boot企业工程成熟度高。 |

## 2. 前端技术决策

### 2.1 默认：React + Ant Design

适用条件：

- 团队具备React或现代前端工程经验。
- V1以Web管理端为主，列表、筛选、表单、弹窗、详情Tab、权限菜单是主要界面形态。
- 需要较快交付企业后台标准控件，不优先自研组件体系。

建议配套：

- React 18+
- Ant Design 5+
- TypeScript
- React Router
- TanStack Query或同类请求状态管理
- 表单优先使用Ant Design Form

### 2.2 可替代：Vue + Element Plus

采用条件：

- 研发团队Vue经验显著强于React。
- 公司已有Vue后台工程模板、权限菜单、请求封装和组件规范。
- 业务方更关注快速复用既有内部后台能力。

替代影响：

- API设计不受影响。
- 页面和组件任务拆分仍按V1页面清单执行。
- 前端任务估算需按团队已有脚手架重新校准。

## 3. 后端技术决策

### 3.1 默认：Java Spring Boot

适用条件：

- CRM需要稳定处理认证、权限、审计、事务、数据权限和业务规则。
- 后续V2会进入合同、开票、回款、核销等强事务场景。
- 团队需要企业级工程结构、分层架构和可维护性。

建议配套：

- Spring Boot 3.x
- Spring Security
- Spring Data JPA或MyBatis Plus，最终由团队编码习惯确认
- Flyway或Liquibase管理DDL版本
- Bean Validation处理参数校验
- OpenAPI文档生成

### 3.2 可替代：Node.js NestJS

采用条件：

- 团队希望前后端TypeScript统一。
- V1交付速度优先，业务规则复杂度初期可控。
- 公司已有NestJS认证、权限和审计基础设施。

决策条件：

- 必须明确事务管理、数据权限拦截、审计日志和后台任务处理方案。
- V2财务核销进入开发前需重新评估事务一致性和工程治理能力。

### 3.3 可替代：Python FastAPI

采用条件：

- AI能力要提前进入V1试点，且团队Python工程能力成熟。
- 后台业务复杂度可控，CRM核心流程优先轻量实现。

决策条件：

- 必须确认权限、审计、迁移、后台任务、测试和部署标准。
- 若V2财务闭环按期进入，应评估是否拆出AI服务而非整体采用FastAPI。

## 4. 数据库决策

### 4.1 默认：PostgreSQL

适用条件：

- 需要 `v_opportunity_weekly_progress` 等视图聚合。
- 审计日志、AI草稿和周进展明细需要JSONB承载结构化快照。
- 后续管理看板可能使用视图、物化视图或复杂分析查询。

V1默认原则：

- 金额：`numeric(18,2)`。
- 比例：`numeric(8,4)`。
- 日期：纯日期使用 `date`，业务时间和审计时间使用 `timestamptz`。
- 枚举：V1使用 `varchar` + 字典表，不先使用数据库原生枚举。
- JSON：使用 `jsonb`。
- 主键：V1 启动默认使用 `bigint generated by default as identity`，所有表统一，不混用 uuid。

### 4.2 可替代：MySQL

采用条件：

- 公司已有MySQL标准、DBA规范、备份监控和运维经验。
- 团队对PostgreSQL缺少生产运维经验。

替代影响：

- JSON字段可使用MySQL JSON，但视图、聚合JSON和日期函数需重新适配。
- 周进展视图实现需验证JSON聚合能力，必要时改为查询服务组装。

## 5. 文件存储决策

默认推荐：对象存储或企业内部文件服务。

V1范围：

- 建立 `crm_attachments` 通用附件表，保存业务对象、文件名、文件地址、类型、上传人、上传时间。
- 后端仅沉淀附件元数据和权限校验，实际文件上传服务根据企业已有设施接入。

待确认：

- 文件服务类型：公有云对象存储、私有化对象存储、企业网盘或内部文件服务。
- 上传方式：后端中转上传或前端直传签名URL。
- 病毒扫描、文件大小、文件类型白名单和下载审计策略。

## 6. 环境决策

默认推荐：测试环境 + 试点生产环境。

| 环境 | 用途 | 最低要求 |
|---|---|---|
| 本地开发 | 前后端日常开发 | 本地数据库或共享开发库、基础种子数据、调试日志。 |
| 测试环境 | 联调、测试用例、回归 | 独立数据库、可重复初始化seed、接口文档、测试账号。 |
| 试点生产环境 | 小范围真实使用 | 备份策略、访问控制、审计日志、基础监控、试点用户名单。 |

待确认：

- 试点生产环境上线日期和试点销售团队范围。
- 是否需要单点登录或企业微信/钉钉/飞书集成。
- 公司基础设施是否已有统一网关、日志、监控和告警平台。

## 7. 决策变更规则

- 若启动会决定采用替代技术栈，应在本文新增“变更记录”，说明变更原因、影响范围、迁移成本和责任人。
- 替代方案不得改变V1业务范围：系统基础、客户、联系人、商机、销售行动、周进展仍为必交付。
- 数据库替代必须同步更新 `docs/database/v1-ddl-plan.md` 中字段类型、JSON、日期函数和视图实现方式。

## 8. 待确认项

- 主键策略：V1 启动默认使用 bigint；如需多系统离线合并或跨库同步，再发起 uuid 变更评审。
- Java持久层选择：JPA、MyBatis Plus或公司既有标准。
- 文件服务类型和上传链路。
- 试点生产环境的安全要求、域名、账号体系和运维责任人。
- 是否接入SSO；V1默认支持账号密码登录，SSO仅预留登录账号类型。
