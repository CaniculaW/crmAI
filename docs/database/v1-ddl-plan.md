# CRM V1 DDL计划

日期：2026-06-17

适用范围：CRM V1系统基础与权限管理、客户池、联系人与干系人、商机生命周期、销售行动、商机周进展汇总。

说明：本文不是完整SQL脚本，而是供后端和DBA拆分DDL任务的结构化计划。V1表清单以研发启动计划Task 4为准。

## 1. 全局DDL原则

### 1.1 主键与租户字段

- 主键策略：V1 启动默认统一使用 `bigint generated by default as identity`，不得在 V1 混用 uuid。
- 本计划用 `id` 表示统一主键。
- 核心业务表预留 `tenant_id`；若V1为单组织部署，可先固定默认租户，但字段和索引预留。

### 1.2 通用审计与软删除字段

除纯关联表、视图、登录日志等特别说明外，核心表统一包含：

| 字段 | 类型建议 | 说明 |
|---|---|---|
| tenant_id | bigint | 租户或组织ID，单组织部署可预留默认值。 |
| created_by | bigint nullable | 创建人，关联 `sys_users.id`。 |
| created_at | timestamptz | 创建时间。 |
| updated_by | bigint nullable | 更新人，关联 `sys_users.id`。 |
| updated_at | timestamptz | 更新时间。 |
| deleted_at | timestamptz nullable | 软删除时间。 |
| version | int | 乐观锁版本。 |

软删除原则：

- 业务主表和字典表使用软删除。
- 关联表可按业务需要硬删除或软删除；若需要审计协同关系变化，则通过审计日志记录。
- 唯一约束需考虑软删除，PostgreSQL建议使用部分唯一索引，例如 `where deleted_at is null`。

### 1.3 字段类型原则

- 金额字段：`numeric(18,2)`。
- 比例字段：`numeric(8,4)`。
- 纯日期字段：`date`。
- 业务时间、登录时间、审计时间：`timestamptz`。
- 枚举字段：`varchar(32/64/128)` + `sys_dict_types/sys_dict_items` 管理，不使用数据库原生enum。
- JSON字段：PostgreSQL使用 `jsonb`，用于审计前后快照、周进展聚合明细和后续AI草稿结构。
- 长文本：`text`。

### 1.4 索引原则

- 所有外键列建立普通索引。
- 高频列表筛选字段建立组合索引，优先覆盖负责人、部门、状态、阶段、时间范围。
- 审计与日志表按操作者、对象、时间建立索引。
- 默认列表不展示软删除数据，常用索引需结合 `deleted_at is null`。

## 2. V1表与视图清单

V1必须覆盖：

- 系统基础：`sys_departments`、`sys_users`、`sys_roles`、`sys_user_roles`、`sys_login_accounts`、`sys_user_credentials`、`sys_sessions`、`sys_password_reset_tokens`、`sys_permissions`、`sys_role_permissions`、`sys_data_scopes`、`sys_role_data_scopes`、`sys_login_logs`、`sys_audit_logs`、`sys_dict_types`、`sys_dict_items`
- CRM业务：`crm_accounts`、`crm_account_collaborators`、`crm_contacts`、`crm_contact_project_roles`、`crm_opportunities`、`crm_opportunity_collaborators`、`crm_opportunity_contacts`、`crm_sales_activities`、`crm_activity_contacts`、`crm_activity_participants`、`crm_activity_risk_types`、`crm_attachments`、`crm_reminders`
- 视图：`v_opportunity_weekly_progress`

## 3. 系统基础与权限表

| 表 | 主键 | 外键 | 唯一约束 | 关键索引 | 关键字段 |
|---|---|---|---|---|---|
| sys_departments | id | parent_id -> sys_departments.id；manager_user_id -> sys_users.id | tenant_id + code | parent_id、manager_user_id、status | name、code、region_code、status |
| sys_users | id | department_id -> sys_departments.id | tenant_id + mobile；tenant_id + email可配置唯一 | department_id、status、role_code | name、mobile、email、role_code、last_login_at、force_password_change |
| sys_roles | id | 无 | tenant_id + code | code、deleted_at | code、name、description |
| sys_user_roles | 复合PK(user_id, role_id) | user_id -> sys_users.id；role_id -> sys_roles.id | PK即唯一 | role_id | 角色分配关系 |
| sys_login_accounts | id | user_id -> sys_users.id | login_type + login_identifier | user_id、status、locked_until | login_type、login_identifier、is_primary、failed_login_count |
| sys_user_credentials | id | user_id -> sys_users.id | user_id + credential_type | user_id、status | password_hash、password_algo、password_updated_at、password_expires_at |
| sys_sessions | id | user_id -> sys_users.id；login_account_id -> sys_login_accounts.id | session_token_hash | user_id、expires_at、revoked_at | session_token_hash、ip_address、user_agent、expires_at、revoked_at |
| sys_password_reset_tokens | id | user_id -> sys_users.id；created_by -> sys_users.id | token_hash | user_id、expires_at、used_at | reset_channel、expires_at、used_at |
| sys_permissions | id | parent_id -> sys_permissions.id | permission_code | module_code、permission_type、parent_id | permission_code、permission_name、permission_type、module_code、sort_order、is_active |
| sys_role_permissions | 复合PK(role_id, permission_id) | role_id -> sys_roles.id；permission_id -> sys_permissions.id | PK即唯一 | permission_id | 角色权限关系 |
| sys_data_scopes | id | 无 | scope_code | scope_code | own、department、department_tree、collaborated、global、custom |
| sys_role_data_scopes | 复合PK(role_id, module_code, data_scope_id) | role_id -> sys_roles.id；data_scope_id -> sys_data_scopes.id | PK即唯一 | module_code、data_scope_id | module_code |
| sys_login_logs | id | user_id -> sys_users.id nullable | 无 | user_id + occurred_at、login_identifier、event_type | event_type、success、failure_reason、ip_address、user_agent、occurred_at |
| sys_audit_logs | id | actor_user_id -> sys_users.id nullable | 无 | actor_user_id + occurred_at、object_type + object_id、module_code + occurred_at | action_code、before_data jsonb、after_data jsonb、result |
| sys_dict_types | id | 无 | dict_code | dict_code、is_active | dict_code、dict_name、is_active |
| sys_dict_items | id | dict_type_id -> sys_dict_types.id | dict_type_id + item_code | dict_type_id + sort_order、is_active | item_code、item_name、sort_order、is_active |

约束补充：

- `sys_login_accounts.login_type` 建议覆盖 `username/mobile/email/sso`。
- `sys_user_credentials.password_hash` 不允许保存明文密码。
- `sys_audit_logs.before_data` 和 `after_data` 使用 `jsonb`。
- 字典项删除优先停用，不建议物理删除。

## 4. 客户域表

### 4.1 crm_accounts

- 主键：`id`
- 外键：`parent_id -> crm_accounts.id`、`owner_department_id -> sys_departments.id`、`owner_user_id -> sys_users.id`
- 唯一约束：启用 `tenant_id + account_name` 软删除唯一；PostgreSQL 使用部分唯一索引 `unique (tenant_id, account_name) where deleted_at is null`。保存前仍提供重复客户提示。
- 关键索引：
  - `owner_user_id`
  - `owner_department_id`
  - `parent_id`
  - `account_level + account_status`
  - `industry + region_province + region_city`
  - `last_activity_at`
- 关键字段：
  - 枚举：account_type、account_level、account_status、account_source、relationship_status
  - 日期：last_activity_at、next_follow_up_at
  - 文本：background、key_needs、last_activity_summary、remark

### 4.2 crm_account_collaborators

- 主键：`(account_id, user_id)`
- 外键：`account_id -> crm_accounts.id`、`user_id -> sys_users.id`
- 索引：`user_id`、`collaborator_role`
- 字段：`collaborator_role`，建议字典值为 sales、presales、delivery、manager
- 用途：支撑客户协同数据权限。

## 5. 联系人与干系人表

### 5.1 crm_contacts

- 主键：`id`
- 外键：`account_id -> crm_accounts.id`
- 唯一约束：启用 `tenant_id + account_id + name + mobile_normalized` 软删除唯一；`mobile_normalized` 为空时不参与强唯一，仅提供重复提示。
- 关键索引：
  - `account_id`
  - `contact_type`
  - `attitude`
  - `relationship_heat`
  - `last_communication_at`
- 关键字段：
  - 枚举：contact_type、decision_influence、attitude、relationship_heat、importance_level
  - 日期：birthday、anniversary、last_communication_at
  - 文本：next_action、last_communication_summary、remark

### 5.2 crm_contact_project_roles

- 主键：`(contact_id, role_code)`
- 外键：`contact_id -> crm_contacts.id`
- 索引：`role_code`
- 字段：`role_code`，建议字典值为 demand_owner、budget_promoter、technical_gatekeeper、procurement_executor、contract_approver、acceptance_owner
- 用途：一个联系人可拥有多个项目角色。

## 6. 商机域表

### 6.1 crm_opportunities

- 主键：`id`
- 外键：`account_id -> crm_accounts.id`、`owner_user_id -> sys_users.id`、`closed_by -> sys_users.id`
- 唯一约束：启用 `tenant_id + account_id + opportunity_name` 软删除唯一；PostgreSQL 使用部分唯一索引 `unique (tenant_id, account_id, opportunity_name) where deleted_at is null`。
- 关键索引：
  - `account_id`
  - `owner_user_id`
  - `stage + status`
  - `expected_close_date`
  - `risk_status`
  - `last_activity_at`
- 关键字段：
  - 金额：estimated_budget_amount、estimated_contract_amount
  - 比例：estimated_gross_margin_rate、win_rate
  - 日期：expected_close_date、last_activity_at、closed_at
  - 枚举：stage、status、level、source、technical_solution_status、stakeholder_plan_status、quotation_status、bid_self_check_status、risk_status、close_type
  - 文本：potential_point、opportunity_analysis、project_requirement、current_progress、next_plan、risk_description、close_description
- 业务约束：
  - `status in ('closed','cancelled')` 时必须有 close_reason、closed_at、closed_by。
  - 默认跟进列表过滤 `status = 'following'` 且 `deleted_at is null`。
  - 周进展不得落在商机主表字段中。

### 6.2 crm_opportunity_collaborators

- 主键：`(opportunity_id, user_id)`
- 外键：`opportunity_id -> crm_opportunities.id`、`user_id -> sys_users.id`
- 索引：`user_id`、`collaborator_role`
- 用途：支撑商机协同和数据权限。

### 6.3 crm_opportunity_contacts

- 主键：`(opportunity_id, contact_id)`
- 外键：`opportunity_id -> crm_opportunities.id`、`contact_id -> crm_contacts.id`
- 索引：`contact_id`、`role_in_opportunity`、`is_key_person`
- 字段：role_in_opportunity、is_key_person
- 用途：支撑商机决策链和关键角色缺口查询。

## 7. 销售行动域表

### 7.1 crm_sales_activities

- 主键：`id`
- 外键：`account_id -> crm_accounts.id`、`opportunity_id -> crm_opportunities.id nullable`、`owner_user_id -> sys_users.id`
- 关键索引：
  - `account_id`
  - `opportunity_id`
  - `owner_user_id + activity_time`
  - `activity_status + activity_time`
  - `next_follow_up_at`
  - `include_in_weekly_progress + activity_status`
- 关键字段：
  - 日期：activity_time、next_follow_up_at
  - 枚举：activity_type、activity_status、activity_result、weekly_period、source_type
  - 文本：communication_content、customer_feedback、conclusion、next_plan、risk_description、remark
  - 布尔：include_in_weekly_progress
  - 预留：source_ai_draft_id，V1可先为空；若未创建AI表，不启用外键。
- 业务约束：
  - `account_id` 必填，`opportunity_id` 可选。
  - 完成行动时回写客户最近跟进；关联商机时回写商机最近跟进。
  - `activity_result = 'risk_found'` 时可触发商机风险状态升级。

### 7.2 crm_activity_contacts

- 主键：`(activity_id, contact_id)`
- 外键：`activity_id -> crm_sales_activities.id`、`contact_id -> crm_contacts.id`
- 索引：`contact_id`
- 用途：记录行动涉及的客户联系人。

### 7.3 crm_activity_participants

- 主键：`(activity_id, user_id)`
- 外键：`activity_id -> crm_sales_activities.id`、`user_id -> sys_users.id`
- 索引：`user_id`
- 用途：记录我方参与人员，并支持“本人参与”数据权限。

### 7.4 crm_activity_risk_types

- 主键：`(activity_id, risk_type)`
- 外键：`activity_id -> crm_sales_activities.id`
- 索引：`risk_type`
- 字段：risk_type，建议字典值为 budget、relationship、competition、technical、procurement、delivery、contract、payment
- 用途：一条行动可标记多个风险类型。

## 8. 通用附件与提醒表

### 8.1 crm_attachments

- 主键：`id`
- 外键：`uploaded_by -> sys_users.id`
- 逻辑关联：`object_type + object_id` 指向业务对象，不做跨表外键。
- 关键索引：`object_type + object_id`、`uploaded_by + uploaded_at`
- 关键字段：
  - 枚举：object_type、file_type
  - 文件字段：file_name、file_url
  - 日期：uploaded_at
- 约束：object_type V1至少覆盖 account、contact、opportunity、activity。

### 8.2 crm_reminders

- 主键：`id`
- 外键：`assignee_id -> sys_users.id`
- 逻辑关联：`object_type + object_id`
- 关键索引：`assignee_id + due_at`、`status + due_at`、`object_type + object_id`
- 关键字段：
  - 枚举：object_type、reminder_type、status
  - 日期：due_at、completed_at
- 用途：销售行动下次跟进、逾期待办、后续审批预留。

## 9. 周进展视图

### 9.1 v_opportunity_weekly_progress

来源表：`crm_sales_activities`

过滤条件：

- `opportunity_id is not null`
- `activity_status = 'completed'`
- `include_in_weekly_progress = true`
- `deleted_at is null`

建议字段：

| 字段 | 类型建议 | 来源或计算 |
|---|---|---|
| opportunity_id | 主键同类型 | crm_sales_activities.opportunity_id |
| account_id | 主键同类型 | crm_sales_activities.account_id |
| owner_user_id | 主键同类型 | crm_sales_activities.owner_user_id |
| week_start_date | date | activity_time按自然周计算 |
| week_end_date | date | week_start_date + 6天 |
| activity_count | bigint | count(*) |
| latest_activity_at | timestamptz | max(activity_time) |
| progress_items | jsonb | 聚合subject、activity_time、conclusion、next_plan、risk_description、activity_result |

索引依赖：

- `crm_sales_activities(opportunity_id, activity_time)`
- `crm_sales_activities(activity_status, include_in_weekly_progress, activity_time)`

实现要求：

- 视图只读，不提供手工编辑周进展接口。
- 同一商机同一自然周多条行动保留多条明细，放入 `progress_items`。
- 商机主表不得新增第一周、第二周、第三周、第四周进展字段。

## 10. DDL拆分建议

1. 批次1：系统基础主表和字典表。
2. 批次2：权限关系表、登录会话、日志和审计表。
3. 批次3：客户、联系人、商机主表。
4. 批次4：协同关系、商机联系人、销售行动及行动关联表。
5. 批次5：附件、提醒、周进展视图。
6. 批次6：补充唯一约束、组合索引、软删除部分唯一索引。

## 11. 待确认项

- 是否 V1 启用 `tenant_id` 多租户隔离，或仅作为字段预留。
- `source_ai_draft_id` 在 V1 是否保留无外键字段，等待 V4 创建 AI 表后再加外键。
- 文件服务接入方式决定 `crm_attachments.file_url` 保存 URL、对象 Key 还是内部文件 ID。
- 联系人手机号为空时的重复提示规则是否需要按姓名、职务、邮箱辅助判断。
