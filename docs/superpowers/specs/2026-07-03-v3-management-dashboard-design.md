# V3 经营驾驶舱设计说明

日期：2026-07-03

分支：`codex/v3-management-dashboard`

## 1. 背景

V1 已完成销售基础闭环，覆盖客户、联系人、商机、销售行动和周进展。V2 已完成销售到财务闭环，覆盖方案标书、合同、开票、回款、到账和发票回款核销。

V3 的目标不是新增另一批录入页面，而是把 V1/V2 的过程数据转成管理层能理解的经营驾驶舱：看总体经营健康度、看销售预测、看合同执行、看开票回款、看风险，并能下钻回到具体业务对象。

## 2. 用户与使用场景

核心用户：

- 管理层：关注收入预测、合同资产、开票进度、回款风险和经营异常。
- 销售负责人：关注销售漏斗、商机阶段、停滞商机、重点客户和团队跟进质量。
- 商务/法务负责人：关注合同状态、节点履约、变更和合同风险。
- 财务负责人：关注应开、已开、应收、实收、逾期、核销和未分配回款。

典型场景：

- 每周经营例会前，管理层打开驾驶舱查看本周经营健康度。
- 销售负责人查看商机漏斗和预测偏差，定位停滞商机。
- 财务负责人查看逾期回款和未核销金额，分派跟进行动。
- 管理者从风险列表下钻到客户、商机、合同、发票、回款或核销明细。

## 3. 范围

V3 做：

- 经营驾驶舱总览。
- 销售漏斗与商机预测。
- 合同看板。
- 开票看板。
- 回款看板。
- 风险预警。
- 数据下钻。
- 权限、审计、测试数据和验收证据。

V3 不做：

- 不做 V4 AI 销售助手。
- 不做 BI 自定义报表设计器。
- 不做外部系统实时集成。
- 不做复杂多维数据仓库。
- 不修改 V1/V2 已验证通过的核心录入和状态流转。

## 4. 方案选择

推荐采用方案 B：先做「经营驾驶舱总览 + 下钻框架」，再逐步补齐各专题看板。

| 方案 | 内容 | 优点 | 风险 |
|---|---|---|---|
| A | 逐个做专题看板：漏斗、预测、合同、开票、回款、风险 | 模块边界清楚 | 第一阶段缺少管理层总览入口 |
| B | 先做总览和下钻框架，再做专题看板 | 符合管理驾驶舱使用习惯，后续模块可自然接入 | 第一模块需要先定义核心指标 |
| C | 先做指标平台/报表配置，再配置驾驶舱 | 长期灵活 | 首版成本高，偏离项目型 CRM 主线 |

选择 B 的原因：

- V1/V2 已有足够明细数据，V3 应优先解决“看全局、找风险、能下钻”。
- 总览页会成为后续专题看板的统一入口。
- 先不做复杂 BI 配置，降低 V3 首版交付风险。

## 5. 页面与导航

新增一级菜单：`驾驶舱`

推荐子页面：

- `经营总览`：默认页，展示关键指标、趋势摘要、风险摘要和待处理排行。
- `销售漏斗`：展示商机阶段分布、转化、预测金额和停滞商机。
- `合同看板`：展示合同金额、状态、节点、变更和履约风险。
- `开票看板`：展示应开、已开、待开、异常、签收和作废。
- `回款看板`：展示应收、实收、逾期、到账、核销和未分配。
- `风险预警`：集中展示商机、合同、开票、回款和核销风险。

首版页面结构：

```text
驾驶舱
├─ 经营总览
│  ├─ 顶部筛选：时间范围、组织、负责人、客户、业务线
│  ├─ 核心指标：预测金额、合同金额、已开票、已回款、逾期金额、风险数
│  ├─ 经营链路：商机 -> 合同 -> 开票 -> 回款 -> 核销
│  ├─ 风险摘要：停滞商机、合同节点逾期、开票异常、回款逾期、核销异常
│  └─ 下钻入口：跳转到对应明细页面并带筛选条件
├─ 销售漏斗
├─ 合同看板
├─ 开票看板
├─ 回款看板
└─ 风险预警
```

## 6. 指标口径

首版核心指标：

| 指标 | 口径 |
|---|---|
| 预测金额 | 未关闭商机的预计金额，可按阶段和预计成交日期过滤 |
| 赢单金额 | 已关闭且关闭类型为赢单的商机金额 |
| 合同金额 | 合同台账中有效合同金额，排除已终止或作废记录 |
| 已开票金额 | 发票状态为已开具或已签收的发票金额 |
| 应收金额 | 回款计划中当前筛选范围内的计划金额 |
| 已回款金额 | 已确认到账的回款金额 |
| 逾期金额 | 到期日早于当前日期且未完成回款的计划剩余金额 |
| 已核销金额 | 有效核销记录的核销金额 |
| 未核销回款 | 已到账回款金额减去已核销金额 |
| 风险数 | 商机停滞、合同节点逾期、开票异常、回款逾期、核销异常的合计数量 |

时间口径：

- 默认看当前自然月，也支持本季度、本年和自定义范围。
- 金额趋势按月聚合。
- 回款逾期按计划 due date 计算。

权限口径：

- 指标统计必须遵守当前用户的数据权限。
- 管理层可看授权组织范围。
- 销售只能看自己负责或协作的数据。
- 财务只能看授权范围内的开票、回款、核销数据。

## 7. API 设计方向

首版后端新增 `dashboard` 模块，不新建业务主表，优先基于现有 V1/V2 表聚合。

推荐 API：

- `GET /api/dashboard/overview`：经营总览指标、链路摘要、风险摘要。
- `GET /api/dashboard/funnel`：销售漏斗、阶段分布、预测金额和停滞商机。
- `GET /api/dashboard/contracts`：合同金额、状态分布、节点风险。
- `GET /api/dashboard/invoices`：开票金额、状态分布、异常清单。
- `GET /api/dashboard/receivables`：应收、实收、逾期、到账、核销摘要。
- `GET /api/dashboard/risks`：统一风险列表和下钻目标。

通用查询参数：

- `date_from`
- `date_to`
- `department_id`
- `owner_id`
- `account_id`
- `opportunity_id`

## 8. 前端设计方向

沿用当前 React + Ant Design + lucide-react 体系。

首版不引入重量级 BI 框架。图表优先使用轻量、可测试的 SVG/CSS 或已有依赖；如果后续需要复杂图表，再评估图表库。

页面原则：

- 后台系统风格，信息密度适中，避免营销式大图。
- 指标卡用于关键经营数字。
- 趋势和漏斗用于判断方向。
- 风险列表用于进入行动。
- 所有图表和列表都必须能下钻到已有业务页面。
- 空数据时给出业务含义，而不是只显示空表。

- 下钻链接测试。

- 管理员登录后能看到驾驶舱菜单。

- 点击风险或指标能下钻到对应业务页面。
- 控制台无应用错误。
