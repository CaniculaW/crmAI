# V1 系统管理复核页面逻辑设计

日期：2026-06-28

## 目标

把当前扁平的系统管理入口调整为专业后台常见的信息架构：左侧一级菜单为“系统”，下挂系统概览、组织管理、用户管理、角色权限、审计日志、字典管理六个子页面。系统概览只承担入口聚合和状态总览，具体维护动作下沉到对应独立页面。

## 当前问题

- 左侧菜单把“系统管理、组织管理、用户管理、角色权限、审计日志”并列为一级菜单，层级不清。
- 字典配置留在系统管理概览页，缺少独立入口，不利于管理员按配置项维护。
- `SystemPage` 同时承载概览、组织、用户、角色、审计和字典表单，页面职责过重。
- “系统管理”既像一级模块名，又像概览页名，容易和子能力混淆。

## 方案 B：系统一级菜单 + 独立子页面

采用以下菜单结构：

```text
系统
├─ 系统概览
├─ 组织管理
├─ 用户管理
├─ 角色权限
├─ 审计日志
└─ 字典管理
```

## 页面职责

| 页面 | 路由 | 职责 |
|---|---|---|
| 系统概览 | `/system` | 展示系统治理入口、组织/用户/角色/字典/审计数量、最近审计摘要和管理提醒，不直接承载大量编辑动作 |
| 组织管理 | `/system/departments` | 维护组织列表、新建组织、组织编码、区域、上级组织和启停状态 |
| 用户管理 | `/system/users` | 维护用户列表、新增/编辑用户、绑定组织和角色、重置密码、启停状态 |
| 角色权限 | `/system/roles` | 查看角色列表、权限点数量、进入授权弹窗维护角色权限 |
| 审计日志 | `/system/audit-logs` | 查看操作时间、模块、动作、对象、操作人、结果和 Trace |
| 字典管理 | `/system/dictionaries` | 维护字典类型、字典项、新增项、编辑项和启停状态 |

## 信息架构要求

- 左侧一级菜单只展示一个“系统”入口；组织、用户、角色权限、审计日志、字典管理作为系统子菜单。
- 一级“系统”需要根据当前用户是否拥有任一系统权限展示。
- 子菜单按权限展示：组织/用户依赖用户管理权限，角色权限依赖角色管理权限，审计日志依赖审计读取权限，字典管理依赖字典管理权限。
- 当前路由在子菜单中保持选中状态，系统概览在 `/system` 选中。
- 字典管理从系统概览中拆出为独立页面，概览页保留“字典管理”入口和数量，不再直接铺开所有字典项。

## 交互要求

- 系统概览提供六个管理入口卡片，其中当前页面不重复跳转自己；其他卡片可进入对应子页面。
- 组织管理保留“新建组织”主动作。
- 用户管理保留“新增用户”和“重置密码”主动作。
- 角色权限保留“授权”动作。
- 审计日志以查询和追溯为主，不提供新增动作。
- 字典管理保留“新建字典”“新增项”“编辑字典项”动作。
- 所有页面保留当前 `DataWorkspace` 的标题、副标题、操作指引、刷新、错误提示和空状态模式。

## 数据与接口

本模块不调整后端模型和接口，继续复用现有接口：

- `crmApi.departments`
- `crmApi.users`
- `crmApi.roles`
- `crmApi.permissions`
- `crmApi.auditLogs`
- `crmApi.dictionaries`
- `resetPassword`

## 不做范围

- 不新增组织审批流。
- 不新增 SSO、MFA、LDAP 等企业集成。
- 不新增复杂权限表达式引擎。
- 不调整审计日志归档或外部 SIEM 集成。
- 不扩大 V1 到合同、开票、回款、经营驾驶舱等 V2 能力。

## 验收点

- 左侧菜单展示“系统”一级菜单，并能展开/进入系统子页面。
- 系统子菜单包含系统概览、组织管理、用户管理、角色权限、审计日志、字典管理。
- `/system/dictionaries` 可独立维护字典类型和字典项。
- `/system` 不再直接承担字典项批量维护，只展示系统入口和状态摘要。
- 组织、用户、角色权限、审计日志页面保持当前可用能力。
- 前端测试覆盖菜单层级、字典独立页面、系统概览入口和现有系统管理动作。
- 本地 UAT 页面无服务端异常横幅，无控制台应用错误。
