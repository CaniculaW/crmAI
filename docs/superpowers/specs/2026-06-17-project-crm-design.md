# 项目型大客户CRM系统设计文档

日期：2026-06-17

## 1. 设计目标

本系统面向项目型大客户销售场景，覆盖从客户经营、机会发现、商机推进、销售行动、方案标书、合同签署、开票、回款到经营分析的完整闭环。

系统不是简单替代Excel表格，而是将销售过程沉淀为可协同、可追踪、可预警、可分析的CRM业务系统。核心目标包括：

- 建立统一客户池，沉淀客户、联系人、干系人和关系经营信息。
- 建立商机生命周期，支撑项目型销售从线索到成交的阶段推进。
- 将销售行动独立成模块，作为客户经营、商机推进和周进展汇总的数据来源。
- 将方案、标书、报价、测算和评审纳入统一过程管理。
- 打通合同、开票、回款和发票核销闭环。
- 建立经营驾驶舱，支持管理层查看漏斗、预测、合同、开票、回款和风险。
- 增加AI销售助手，帮助业务人员完成信息识别、草稿生成、补全追问和确认入库。

## 2. 总体模块架构

系统由十个核心模块组成：

1. 客户池
2. 联系人与干系人
3. 商机生命周期
4. 销售行动
5. 方案与标书
6. 合同管理
7. 开票管理
8. 回款管理
9. 经营驾驶舱
10. AI销售助手

主业务链路为：

客户池 -> 联系人与干系人 -> 商机生命周期 -> 销售行动 -> 方案与标书 -> 合同管理 -> 开票管理 -> 回款管理 -> 经营驾驶舱

AI销售助手是横向交互层，服务所有业务模块，不替代业务模块本身。

## 3. 关键设计原则

- 客户是主对象，商机是销售主线，销售行动是过程证据。
- 商机阶段只表达推进进度，商机状态表达是否继续推进。
- 商机表不作为主要周报录入入口，周进展由销售行动自动汇总生成。
- 合同、开票、回款拆分为独立模块，便于销售、商务、财务协同。
- 回款与发票采用多对多核销模型，支持一笔回款核销多张发票，也支持一张发票分多笔回款核销。
- AI销售助手生成草稿和建议，关键业务数据必须经人工确认后写入。
- 所有关键状态变更、金额变更和AI写入动作必须可追溯。

## 4. 客户池模块

### 4.1 模块定位

客户池用于管理目标客户、存量客户、集团客户及其下属单位，是CRM主数据入口。客户池主要回答三个问题：客户是谁、归谁负责、是否值得持续经营。

### 4.2 核心功能

- 客户建档
- 集团与子公司关系管理
- 客户分级
- 行业标签
- 区域管理
- 客户归属管理
- 客户画像
- 重复客户合并
- 最近跟进信息自动回写

## 5. 联系人与干系人模块

### 5.1 模块定位

联系人与干系人模块不只是通讯录，而是客户关系地图。它帮助销售判断项目中谁说了算、谁影响决策、谁支持或阻碍，以及下一步应该经营谁。

### 5.2 核心功能

- 联系人档案
- 客户组织关系
- 决策链管理
- 项目角色管理
- 影响力与态度标记
- 关系热度管理
- 关键人跟进
- 沟通历史沉淀
- 关系缺口识别

### 5.3 关系视图

- 客户关系地图：按客户展示所有联系人、部门、职务、影响力、态度、关系热度、关键程度。
- 商机决策链视图：按商机展示决策人、使用人、技术负责人、采购负责人、财务负责人、外部伙伴。
- 关系态度视图：按支持、中立、观望、反对、未知分组。
- 关系缺口视图：提示缺决策人、缺采购负责人、缺技术负责人、缺财务/合同审批人、关键人长期未跟进、反对者未制定经营动作等问题。

## 6. 商机生命周期模块

### 6.1 模块定位

商机生命周期承接项目型销售主线，用于管理从机会发现到商业成交的全过程。

### 6.2 商机阶段

商机阶段表示业务推进到哪一步：

1. 商业线索
2. 商业验证
3. 商业方案
4. 商业谈判
5. 商业成交

### 6.3 商机状态

商机状态表示商机是否继续推进：

| 状态 | 含义 |
|---|---|
| 跟进中 | 正常推进，可创建销售行动、待办、提醒 |
| 暂停 | 暂时不推进，未来可能重启 |
| 关闭 | 商机归档，不再进入销售预测 |
| 取消跟进 | 明确放弃继续投入，关闭待办和提醒 |

关闭或取消跟进时，需要记录关闭类型、关闭原因、关闭说明、关闭时间、关闭操作人、是否可重启。

### 6.4 核心功能

- 商机登记
- 阶段推进
- 状态管理
- 预算与合同金额管理
- 采购流程记录
- 赢率预测
- 风险管理
- 阶段准入与退出规则
- 最近跟进自动回写
- 周进展汇总视图

### 6.5 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| 商机名称 | 文本，必填 | 项目或机会名称 |
| 所属客户 | 关联客户，必填 | 来源客户 |
| 关联联系人/干系人 | 多关联 | 项目相关人员 |
| 商机阶段 | 单选，必填 | 商业线索、商业验证、商业方案、商业谈判、商业成交 |

| 商机来源 | 单选 | 客户提出、政策资金、伙伴推荐、招投标、存量经营、市场活动 |

| 备注 | 长文本 | 补充信息 |
| 附件 | 文件 | 需求资料、方案、纪要等 |

### 6.6 周进展规则

商机表不作为主要周报录入入口。周进展由销售行动自动汇总生成，并在商机详情页以周进展汇总视图呈现。

汇总规则：

- 只汇总已完成的销售行动。
- 按行动时间、行动状态、关联商机生成周进展。
- 优先取销售行动中的形成结论、下一步计划、风险说明。
- 同一商机一周多条行动时，保留多条进展明细，不强行合并。
- 可按本周、本月、第一周、第二周、第三周、第四周展示。

### 6.7 阶段规则

- 商业线索：必须有客户、机会点、责任人。
- 商业验证：必须补充项目需求、预算、周期、采购流程。
- 商业方案：必须有技术方案状态、干系人方案状态、当前进展。
- 商业谈判：必须有报价/利润测算、标书自评、预计合同金额。
- 商业成交：进入合同管理，生成或关联合同记录。
- 商机状态为关闭或取消跟进后，不再出现在默认跟进列表，不再计入销售漏斗预测金额。

## 7. 销售行动模块

### 7.1 模块定位

销售行动记录销售为了推进客户和商机做了什么、下一步要做什么、是否逾期、是否存在风险。它可以关联客户、联系人和商机，但不要求每条行动都必须关联商机。

### 7.2 核心功能

- 拜访记录
- 会议记录
- 电话、微信、邮件沟通记录
- 方案汇报记录
- 回款跟进记录
- 周进展自动汇总
- 下一步计划
- 待办与提醒
- 行动复盘
- 风险跟进

### 7.3 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| 行动主题 | 文本，必填 | 行动标题 |
| 行动类型 | 单选，必填 | 拜访、会议、电话、微信、邮件、方案汇报、客户演示、投标沟通、回款跟进、内部协同 |
| 关联客户 | 关联客户，必填 | 行动所属客户 |
| 关联商机 | 关联商机，可选 | 具体项目推进动作时关联 |
| 关联联系人 | 多关联 | 涉及的客户联系人 |
| 责任人 | 用户，必填 | 行动负责人 |
| 参与人员 | 多用户 | 我方参与人员 |
| 行动时间 | 日期时间，必填 | 计划或实际发生时间 |
| 行动状态 | 单选，必填 | 待执行、已完成、已取消、已延期 |
| 行动结果 | 单选 | 有推进、无明显推进、待客户反馈、发现风险、完成关键节点 |
| 沟通内容 | 长文本 | 沟通过程 |
| 客户反馈 | 长文本 | 客户态度、关注点、异议 |
| 形成结论 | 长文本 | 本次行动形成的判断 |
| 下一步计划 | 长文本 | 后续动作 |
| 下次跟进时间 | 日期时间 | 生成提醒 |
| 风险类型 | 多选 | 预算、关系、竞争、技术、采购、交付、合同、回款 |
| 风险说明 | 长文本 | 风险详情 |
| 附件 | 文件 | 会议纪要、照片、材料、录音转文字等 |
| 是否进入周进展 | 是/否 | 是否纳入商机周进展汇总 |
| 周进展周期 | 单选/日期 | 自然周或月内第几周 |
| 创建人 | 系统字段 | 系统生成 |
| 创建时间 | 系统字段 | 系统生成 |
| 更新时间 | 系统字段 | 系统生成 |

### 7.4 规则

- 销售行动完成后，自动回写客户最近跟进时间和内容。
- 若关联商机，自动回写商机最近跟进时间和最近跟进摘要。
- 若填写下次跟进时间，自动生成提醒。
- 行动结果为发现风险时，自动更新商机风险状态。
- 商机关闭或取消跟进后，不允许新增普通跟进行动，只允许新增复盘记录或重启申请。

## 8. 方案与标书模块

### 8.1 模块定位

方案与标书模块采用成熟系统中的项目材料台账、协同任务、报价测算、版本审批模式。它不做重型投标系统，但要支撑大客户项目中的需求资料、方案、报价、标书、自评和客户反馈闭环。

### 8.2 核心功能

- 材料台账
- 方案/标书创建
- 版本管理
- 协同任务
- 报价测算
- 利润评审
- 标书自评
- 内部审批
- 客户提交
- 客户反馈
- 商机回写
- 附件归档

### 8.3 状态

| 状态 | 说明 |
|---|---|
| 待编制 | 已创建但未开始 |
| 编制中 | 正在准备材料 |
| 内部评审中 | 提交内部评审 |
| 待修改 | 评审或客户反馈后需修改 |
| 已提交客户 | 已正式提交客户 |
| 客户反馈中 | 等待或处理中客户反馈 |
| 已定稿 | 当前版本确认 |
| 作废 | 版本废弃，不再使用 |

### 8.4 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| 方案/标书名称 | 文本，必填 | 材料名称 |

| 关联商机 | 关联商机，必填 | 所属商机 |
| 类型 | 单选，必填 | 需求资料、技术方案、干系人方案、报价测算、利润测算、投标文件、标书自评、演示材料 |
| 版本号 | 文本 | V1、V2、终版等 |
| 当前状态 | 单选 | 待编制、编制中、内部评审中、待修改、已提交客户、客户反馈中、已定稿、作废 |

### 8.5 规则

- 一笔回款流水可核销多张发票。
- 一张发票可被多笔回款流水分次核销。
- 回款流水已核销金额不能超过到账金额。
- 发票已核销金额不能超过发票金额。
- 核销明细确认后，更新发票已核销金额、回款流水已核销金额、合同已回款金额。
- 回款未完全分配时，状态为待分配或部分核销。
- 发票未完全核销时，状态为部分核销。
- 销售负责回款跟进，财务负责到账确认和核销确认。

### 13.8 AI规则

- 普通销售行动可由AI草稿确认后一键写入。
- 客户、联系人、商机可以AI辅助新建或更新。
- 合同、开票、回款必须由对应角色确认后写入。
- 金额、日期、客户主体、合同主体出现冲突时，必须进入确认任务。
- AI提取置信度低时，不允许直接写入。
- 每一次AI写入都能追溯到原始输入。
- AI生成的周进展来自销售行动，不直接修改商机阶段。
- AI可建议商机阶段推进，但阶段变更需要负责人确认。
- AI不能绕过字段必填、阶段规则、金额校验和权限控制。

## 14. 权限设计

| 角色 | 权限范围 |
|---|---|
| 销售个人 | 查看和维护自己的客户、商机、行动；查看相关合同、开票、回款进度 |
| 销售负责人 | 查看本部门客户、商机、行动、预测、风险和回款情况 |
| 售前/方案人员 | 查看关联客户和商机，维护方案与标书材料 |
| 商务/法务 | 维护合同、审批、风险条款和附件 |
| 财务 | 维护开票、回款流水、发票核销和财务看板 |
| 管理层 | 查看全局经营驾驶舱和重点项目 |
| 系统管理员 | 配置字段、阶段、权限、提醒规则和指标口径 |
| AI销售助手 | 生成草稿、建议和确认任务；不直接绕过人工确认写入关键数据 |

## 16. 后续实施建议

建议按以下顺序实施：

1. 先完成客户池、联系人与干系人、商机生命周期、销售行动四个销售基础模块。
2. 再完成方案与标书、合同管理、开票管理、回款管理四个销售到财务闭环模块。
3. 然后建设经营驾驶舱，基于前面模块汇总指标和预警。
4. 最后接入AI销售助手，从销售行动自动录入和周进展汇总开始，逐步扩展到合同、开票、回款草稿生成。

AI销售助手第一阶段建议优先支持：

- 文本录入销售行动
- 会议纪要生成行动记录
- 自动匹配客户和商机
- 自动提取下一步计划和风险
- 生成商机周进展汇总

合同、开票、回款类AI录入应放在第二阶段，并保留更严格的人工确认和权限校验。
