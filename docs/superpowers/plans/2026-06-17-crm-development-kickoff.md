# 项目型大客户CRM研发启动实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 启动项目型大客户CRM研发，先交付可上线试点的V1销售基础闭环，再分阶段扩展财务闭环、经营驾驶舱和AI销售助手。

**Architecture:** 建议采用模块化单体架构启动，按系统基础、客户、联系人、商机、销售行动、方案合同财务、驾驶舱、AI助手划分业务域。V1优先实现认证权限、客户池、联系人与干系人、商机生命周期、销售行动和商机周进展汇总，确保销售过程先跑起来。

**Tech Stack:** 建议前端采用 Web 管理端优先，后端采用模块化服务端，数据库采用 PostgreSQL 或 MySQL，文件存储采用对象存储或企业文件服务。最终技术栈在启动会中确认后，基于本计划拆解为详细开发任务。

## 2. 研发启动目标

研发启动阶段要完成三件事：

1. 明确第一版上线范围，避免一次性开发完整CRM导致周期过长。
2. 明确技术架构、团队分工、里程碑和验收口径。
3. 形成可进入Sprint执行的任务池，包括产品、设计、前端、后端、测试和AI能力任务。

## 3. MVP范围建议

### 3.1 V1必须交付

V1目标是替代销售Excel中的核心客户、商机、行动和周进展管理。

V1模块：

- 系统基础与权限管理
- 客户池
- 联系人与干系人
- 商机生命周期
- 销售行动
- 商机周进展汇总视图

V1必须跑通的业务闭环：

用户登录 -> 创建客户 -> 创建联系人/干系人 -> 创建商机 -> 推进商机阶段与状态 -> 创建销售行动 -> 自动回写客户和商机最近跟进 -> 自动生成商机周进展汇总

### 3.2 V1暂不交付

以下能力进入后续版本：

- 方案与标书
- 合同管理
- 开票管理
- 回款管理
- 发票与回款多对多核销
- 经营驾驶舱完整指标
- AI销售助手完整能力

### 3.3 V1可预留

V1数据库和接口设计应预留以下扩展点：

- 商机转合同入口
- 销售行动来源字段，支持后续AI写入
- 附件通用对象
- 字典配置
- 审计日志
- 数据权限

## 6. 技术架构建议

### 6.1 推荐启动架构

建议使用模块化单体，而不是一开始拆微服务。

原因：

- 当前系统业务边界清晰，但团队启动阶段更需要快速交付。
- 客户、商机、行动、合同、回款之间关联紧密，过早拆服务会增加事务和联调成本。
- 后续可按模块边界拆出AI助手、报表分析或财务核销服务。

### 6.2 后端模块划分

- Auth模块：登录、登出、会话、密码、用户、角色、权限、审计。
- CRM基础模块：客户、联系人、干系人。
- Opportunity模块：商机阶段、状态、协同人、商机关联联系人。
- Activity模块：销售行动、参与人、风险类型、周进展汇总。
- Document模块：方案与标书，V2启用。
- Contract模块：合同、合同变更、合同节点，V2启用。
- Finance模块：开票、回款计划、回款流水、核销明细，V2启用。
- Dashboard模块：经营驾驶舱，V3启用。
- AI Assistant模块：输入记录、提取草稿、确认任务、写入日志，V4启用。

### 6.3 前端信息架构

- 客户池

- 商机
- 销售行动

- [x] **Step 1: 确认前端技术**

建议选项：

| 选项 | 适用情况 |
|---|---|

建议选项：

| 选项 | 适用情况 |
|---|---|
| Java Spring Boot | 企业系统、权限、事务、复杂业务稳定 |
| Node.js NestJS | 前后端TypeScript统一、迭代快 |
| Python FastAPI | AI能力集成便利、后台业务复杂度适中 |

推荐：若团队无明确偏好，企业CRM优先 Java Spring Boot 或 NestJS。

- [x] **Step 3: 确认数据库**

建议：

- MySQL：适合团队已有MySQL运维经验。

推荐：优先 PostgreSQL；若公司已有MySQL标准，则使用MySQL。

- [x] **Step 4: 写入技术栈决策文档**

技术栈决策文档建议先写入默认推荐方案；若启动会决定采用其他栈，记录替代方案和理由。

```markdown
# CRM技术栈决策

## 前端

默认推荐：React + Ant Design。

理由：企业后台表格、筛选、表单、权限菜单和详情页较多，Ant Design生态成熟。

## 后端

默认推荐：Java Spring Boot。

理由：企业CRM涉及认证权限、事务、审计、复杂业务规则和后续财务核销，Spring Boot工程成熟度高。

## 数据库

默认推荐：PostgreSQL。

理由：支持复杂查询、视图、JSON字段和后续AI草稿结构化数据存储。

## 文件存储

默认推荐：对象存储或企业内部文件服务。

理由：客户资料、方案标书、合同附件、发票附件后续会持续增长，文件与业务数据库分离更稳妥。

## 部署环境

默认推荐：先部署测试环境和试点生产环境。

理由：V1需要业务试点和快速反馈，测试环境用于研发联调，试点生产环境用于小团队真实使用。
```

## 11. V1上线验收标准

V1上线前必须满足：

- 用户可以登录、登出、修改密码。
- 管理员可以维护组织、用户、角色、权限、字典。
- 销售可以创建客户、联系人、商机、销售行动。
- 商机阶段和商机状态可独立维护。
- 商机关闭和取消跟进必须填写原因。
- 销售行动完成后自动回写客户和商机最近跟进。
- 商机周进展由销售行动自动汇总生成。
- 销售负责人可以查看本部门客户、商机、行动。
- 销售个人不能越权查看无关数据。
- 关键操作可以在审计日志中查看。

## 12. 试点上线建议

### 12.1 试点范围

建议先选择一个销售团队试点，人数控制在5-10人。

试点数据：

- 重点客户10-20个
- 联系人30-60个
- 活跃商机20-40个
- 销售行动每人每周至少3条

### 12.2 试点周期

建议试点2周。

试点观察指标：

- 销售行动录入率
- 商机周进展完整率
- 商机最近跟进准确率
- 销售反馈问题数量
- 权限问题数量
- 表单字段调整数量

### 12.3 试点复盘

复盘重点：

- 哪些字段销售不愿填。
- 哪些列表筛选最高频。
- 周进展汇总是否可直接用于销售例会。
- 商机阶段规则是否过重。
- 数据权限是否影响协作。

## 13. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| V1范围膨胀 | 延期且质量下降 | 严格冻结V1，新增需求进入版本池 |
| 销售录入积极性低 | 数据不完整 | V1表单字段保持必要最小，V4引入AI助手 |
| 权限设计过细 | 开发复杂、上线慢 | V1先支持本人、本部门、协同、全局 |
| 周进展口径争议 | 销售例会不可用 | 试点期按销售行动明细展示，不强制合并摘要 |
| 技术栈迟迟不定 | 无法排期 | 启动会必须做技术栈决策 |
| 数据质量差 | 驾驶舱不可用 | V1加入必填校验、重复校验和审计日志 |
| 一次性做AI | 影响主流程交付 | AI助手放V4，V1只预留source字段 |
