# Approval Workflow Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a configurable sequential approval module and connect it to quotation, bid, and contract records with role-based tasks, audit history, and business-state synchronization.

**Architecture:** Add a focused `approval` backend package backed by template, instance, node, and action tables. `ApprovalService` owns workflow transitions and updates business status through JDBC so it does not create circular dependencies with solution and contract services; those services remain responsible for business visibility and edit validation. The frontend adds typed approval APIs, a first-level approval center, a system approval configuration page, and reusable approval panels in solution and contract details.

**Tech Stack:** Java 17, Spring Boot, JdbcTemplate, Flyway, JUnit 5, MockMvc, PostgreSQL/H2 compatibility, React, TypeScript, Ant Design, Vitest, Testing Library.

Create five tables using the established identity, timestamp, soft-delete, and index conventions. The core columns must be:

```sql
create table approval_templates (
    id bigint generated by default as identity primary key,
    tenant_id bigint not null default 1,
    object_type varchar(32) not null,
    template_name varchar(128) not null,
    status varchar(32) not null default 'active',
    is_default boolean not null default false,
    created_by bigint,
    created_at timestamp with time zone not null default current_timestamp,
    updated_by bigint,
    updated_at timestamp with time zone not null default current_timestamp,
    deleted_at timestamp with time zone,
    version integer not null default 0
);

create table approval_template_nodes (
    id bigint generated by default as identity primary key,
    template_id bigint not null references approval_templates(id),
    step_order integer not null,
    node_name varchar(128) not null,
    approver_role_id bigint not null references sys_roles(id),
    status varchar(32) not null default 'active',
    created_at timestamp with time zone not null default current_timestamp,
    updated_at timestamp with time zone not null default current_timestamp
);

create table approval_instances (
    id bigint generated by default as identity primary key,
    tenant_id bigint not null default 1,
    template_id bigint not null references approval_templates(id),
    object_type varchar(32) not null,
    object_id bigint not null,
    object_name varchar(255) not null,
    status varchar(32) not null,
    current_step_order integer,
    submitted_by bigint not null references sys_users(id),
    submitted_at timestamp with time zone not null default current_timestamp,
    completed_at timestamp with time zone,
    result_comment varchar(1000),
    created_at timestamp with time zone not null default current_timestamp,
    updated_at timestamp with time zone not null default current_timestamp
);
```

`approval_instance_nodes` copies `step_order`, `node_name`, and `approver_role_id`, and stores `status`, `handled_by`, `handled_at`, and `comment`. `approval_actions` stores `instance_id`, optional `node_id`, `action`, `actor_user_id`, `comment`, and `action_at`. Add a partial unique index preventing more than one pending instance for the same object, and seed the four permission codes with `module_code = 'approval'`.

- [x] **Step 3: Implement workflow transitions and queries**

Implement these service entry points:

```java
ApprovalInstanceResponse submit(
        String objectType, Long objectId, String objectName, Long actorUserId);
ApprovalInstanceResponse approve(Long instanceId, String comment, Long actorUserId);
ApprovalInstanceResponse reject(Long instanceId, String comment, Long actorUserId);
List<ApprovalTaskResponse> tasks(String bucket, Long actorUserId);
ApprovalInstanceDetailResponse instanceDetail(Long instanceId, Long actorUserId);
ApprovalObjectStatusResponse objectStatus(String objectType, Long objectId, Long actorUserId);
```

Submission copies active template nodes, marks the first node `pending` and the rest `waiting`, writes a `submit` action, and calls the business-state mapper. Approval verifies membership in the current node role through `sys_user_roles`, marks the node approved, advances one node or completes the instance, and writes an action. Rejection requires a nonblank comment, marks the current node and instance rejected, resets the business object to its draft state, and writes an action. All transitions run in a transaction and update rows only when their prior status is still `pending`.
